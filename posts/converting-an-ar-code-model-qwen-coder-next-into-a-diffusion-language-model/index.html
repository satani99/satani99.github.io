    Auto-regressive models like Qwen3-Coder-Next generate code token-by-token, but diffusion-based LMs (dLLMs) generate by denoising an entire sequence in parallel. Recent research suggests diffusion is very promising for code: Apple’s DiffuCoder (7B parameters) is a diffusion LLM fine-tuned on code tasks, built from the Qwen-2.5-Coder backbone【35†L252-L261】. It iteratively masks random tokens and learns to reconstruct them, enabling “global planning” of code rather than strict left-to-right generation【35†L252-L261】【5†L73-L82】. In practice, this can yield competitive or superior pass@k on benchmarks (DiffuCoder beats other code models on MBPP【35†L252-L261】).

    In this post, we show how to adapt Qwen3-Coder-Next (an 80B AR code model) into a diffusion code model. We cover the libraries, data, and methods involved, and give runnable code examples. Throughout, we cite open-source tools and recent papers (with lineage from Qwen and DiffuCoder) to guide the process.

    Libraries and Frameworks#

    dLLM (ZHZisZZ/dllm): A unified framework for diffusion LMs【18†L281-L290】. This library provides training and evaluation pipelines built on Hugging Face Transformers (via Trainer), with support for DeepSpeed, FSDP, LoRA, etc【18†L281-L290】. It even includes Tiny-A2D recipes for converting any AR model (e.g. Qwen, LLaMA, GPT-2) into a diffusion model【18†L300-L303】. In practice, you can pip install or clone this repo and use its templates. For example:

    git clone https://github.com/ZHZisZZ/dllm
cd dllm
pip install -e .

    This gives you commands like dllm-train and configs for masked-diffusion training.

    Open-dLLM / Open-dCoder: An open-source end-to-end stack for code diffusion LMs【29†L346-L354】. It contains data scripts, training configs, inference tools, and checkpoints. Notably, Open-dCoder uses the Qwen2.5-Coder model and FineCode corpus to train a 0.5B diffusion code model【31†L524-L532】. It even provides example training commands. For instance, their README shows how to launch a pretraining run on Qwen2.5-Coder with MDM (masked diffusion):

    torchrun --nnodes=1 --nproc-per-node=4 tasks/train_torch.py \
  configs/pretrain/qwen2_5_coder_500M.yaml \
  --data.train_path=data/fine_code/ \
  --train.micro_batch_size=16 \
  --train.global_batch_size=512 \
  --train.output_dir=logs/qwen2-0.5B_mdm

    This uses 16×32A100 GPUs (global batch 512) and an MDM loss with uniform masking【31†L524-L532】【31†L537-L545】. You could adapt such a command for Qwen3-Coder-Next by pointing to your AR checkpoint and a larger config.

    DiffuLLaMA / DiffuGPT: Research code for adapting GPT-2 and LLaMA to diffusion【23†L281-L290】. While not a ready-to-install package, it contains scripts demonstrating continuous pretraining: load an AR model, change to diffusion loss, train ~1–3 epochs. They report that with <200B tokens, GPT-2/LLama can be fully converted to diffusion models. This code shows principles like adjusting attention masks and diffusion timesteps. You can study it to craft your own adaptation.

    Hugging Face Transformers: Many diffusion LLMs reuse HF interfaces. For example, the Apple DiffuCoder model on Hugging Face can be loaded with:

    from transformers import AutoModel, AutoTokenizer
model = AutoModel.from_pretrained("apple/DiffuCoder-7B-cpGRPO", trust_remote_code=True)
tokenizer = AutoTokenizer.from_pretrained("apple/DiffuCoder-7B-cpGRPO", trust_remote_code=True)

    The custom trust_remote_code=True hook loads their diffusion utilities. Then you can generate with:

    inputs = tokenizer(prompt, return_tensors="pt")
outputs = model.diffusion_generate(
    inputs.input_ids, max_new_tokens=128,
    steps=200, temperature=0.7, top_p=0.95)
print(tokenizer.decode(outputs.sequences[0], skip_special_tokens=True))

    This diffusion_generate API (also used in Open-dLLM and dLLM) handles the iterative denoising internally.

    In summary, the main tools are dLLM, Open-dLLM/Open-dCoder, and Transformers code with diffusion extensions. Installing them typically involves cloning and pip install -e . (for example, dLLM and Open-dLLM instruct to install with pip install -e . as shown in their READMEs【50†L1-L9】【51†L23-L28】).

    AR→Diffusion Conversion Methodology#

    Converting an AR code model into a diffusion model involves several key changes:

    • Masked Diffusion Objective (MDM): We switch from next-token cross-entropy to a masking diffusion loss【43†L194-L202】. In practice, each training batch, we randomly mask some tokens (using a predefined noise schedule) and train the model to predict the original tokens. For example, the Open-dLLM pipeline masks a uniform fraction of tokens in each sequence【31†L524-L532】. Apple’s DiffuCoder paper describes this as an absorbing discrete diffusion process: tokens are gradually turned into a special [MASK] token according to a transition matrix【43†L194-L202】. The model minimizes the ELBO (approximated by cross-entropy) to reconstruct the original sequence.

    • Attention Patterns: A fully bidirectional model might “forget” the AR pretraining, so Efficient-DLM suggests using blockwise causal attention【12†L88-L97】. That is, split the sequence into fixed-size blocks (e.g. 64 tokens), allow full attention within each block, but enforce left-to-right causality between blocks. This preserves the prefix context for inference and lets us cache key/value states. Implementing this is a one-line change in the attention mask (or use an existing config). It’s not always mandatory, but it’s known to retain AR behavior in the diffusion model【12†L88-L97】.

    • Mask Scheduling: You can bias masking toward later tokens (so early code tokens are generated with high confidence). Efficient-DLM’s position-dependent masking sets lower mask probabilities for early positions【12†L88-L97】. Many repositories offer this as a config option. For simplicity, you might start with uniform random masking (like Dream or DiffuCoder) and experiment later.

    • Continual Pretraining: Load your AR model (Qwen-Coder-Next) and simply continue training with the diffusion objective. This is exactly what Apple did: DiffuCoder’s Stage-1 “adaptation pretraining” continues Qwen-2.5-Coder on 400B code tokens with masked diffusion【36†L125-L134】. In pseudocode:

      model = Qwen3ForCausalLM.from_pretrained("Qwen/qwen3-coder-next")
diffusion_trainer = DiffusionTrainer(model=model, mask_schedule=uniform, ...)
diffusion_trainer.train(code_dataset, num_steps=...)

      While specifics depend on the library (dLLM or Open-dLLM handle this pipeline), the idea is to treat it like a new pretraining pass with mask noise.

    • Fine-Tuning / RL: After pretraining, you may fine-tune with instruction-data or RL. Apple’s DiffuCoder did further training: they ran one epoch of coupled-GRPO on 21K code samples (with auto-graded tasks)【41†L51-L57】. This RL step improved DiffuCoder’s pass@k (EvalPlus) by 4.4%. If you have reward signals (e.g. unit tests), you can similarly apply an RL fine-tune using policy gradients (tools like trl or custom GRPO code).

    Data Preparation and Tokenization#

    Code Datasets: Use large, clean code corpora. Some options:

    • FineCode (HF dataset): a concise, high-quality GitHub corpus【31†L524-L532】.
    • RefineCode + The Stack v2: used by DiffuCoder for 400B tokens of code【36†L125-L134】.
    • OpenAI’s instruction datasets (like OpenCoder data, 436K code-comment pairs) for final tuning【36†L125-L134】.
    • Test-case datasets (like AceCoder-87K) if doing RL【36†L125-L134】.
    • For evaluation: HumanEval, MBPP, BigCodeBench (to measure pass@k)【39†L13-L16】.

    Preprocessing:

    • Cleaning: Remove duplicates, filter out very large or trivial files, etc. The exact steps follow what AR code LMs do (the Qwen3-Coder data pipeline is not public, but you can mirror typical code-LLM preprocessing).
    • Subword Tokenization: Use the same tokenizer as Qwen3-Coder-Next (available on Hugging Face) to maintain consistency. In Python:
      from transformers import AutoTokenizer
tokenizer = AutoTokenizer.from_pretrained("Qwen/Qwen3-Coder-Next")
      This ensures that tokens (keywords, identifiers, etc.) match the pretrained vocab. Tokenizer settings (like special tokens for code) are already baked in.
    • Mask Augmentation: For each training example, randomly mask tokens. With dLLM or Open-dLLM, this is handled automatically. If doing custom training, insert a [MASK] token ID for chosen positions. The Apple DiffuCoder framework describes the forward process as replacing tokens with one-hot mask states【43†L194-L202】.

    Training Configuration and Example#

    Below is a simplified example of how one might launch training using a Hugging Face-like interface. This is illustrative; real training (especially at Qwen scale) requires distributed setups.

    from transformers import AutoConfig, AutoModelForMaskedLM, Trainer, TrainingArguments
from datasets import load_dataset
import torch

# 1. Load AR model config & tokenizer
config = AutoConfig.from_pretrained("Qwen/qwen3-coder-next")
tokenizer = AutoTokenizer.from_pretrained("Qwen/qwen3-coder-next")

# 2. Prepare model for diffusion
model = AutoModelForMaskedLM.from_config(config)
model.qwen = load_model("Qwen/qwen3-coder-next", ... )  # pseudo-code: load AR weights
# Replace LM head with diffusion head if needed (could be same, just train with mask)
# Many libraries automate this; shown here for illustration.

# 3. Load code data
dataset = load_dataset("fine_code")  # example FineCode dataset
def tokenize(ex):
    return tokenizer(ex["code"], truncation=True, max_length=1024)
dataset = dataset.map(tokenize, batched=True, remove_columns=["code"])

# 4. Training arguments (masking diff objective done in loss function)
training_args = TrainingArguments(
    output_dir="./dllm-qwen3",
    num_train_epochs=3,
    per_device_train_batch_size=16,
    gradient_accumulation_steps=8,  # to simulate large batch
    fp16=True,
    logging_steps=1000,
    save_steps=10000,
)

# 5. Custom data collator that creates random masks
from transformers import DataCollatorForLanguageModeling
data_collator = DataCollatorForLanguageModeling(
    tokenizer=tokenizer, mlm=True, mlm_probability=0.15
)

# 6. Trainer with mask (diffusion) loss
trainer = Trainer(
    model=model,
    args=training_args,
    data_collator=data_collator,
    train_dataset=dataset["train"],
)

# 7. Train (continual pretraining of AR model with diffusion objective)
trainer.train()

    This code uses HF’s DataCollatorForLanguageModeling as a stand-in for a diffusion mask scheduler. In a real diffusion LM setup, you’d mask multiple times per example with varying rates. Frameworks like dLLM/Open-dLLM internally implement the proper mask scheduling and loss. But the key point is: we load the Qwen model, prepare a masked-LM training run, and continue training.

    After this pretraining, you could optionally continue with RL fine-tuning. For example, using trl:

    from trl import PPOTrainer, PPOConfig

ppo_config = PPOConfig(model_name="Qwen/qwen3-coder-next", learning_rate=1e-6, ...)
ppo_trainer = PPOTrainer(ppo_config, model, tokenizer, dataset=qa_dataset)
losses = ppo_trainer.train()

    This would require setting up a reward (e.g. +1 for correct code execution). The Apple team saw +4.4% on pass@k by doing one RL epoch on 21K examples【41†L51-L57】.

    Inference Example#

    Once trained, you generate with diffusion_generate. For example, to have Qwen3-dLLM generate code:

    from transformers import AutoTokenizer, AutoModel

model = AutoModel.from_pretrained("path/to/trained-diffusion-model", trust_remote_code=True)
tokenizer = AutoTokenizer.from_pretrained("path/to/trained-diffusion-model", trust_remote_code=True)

prompt = "<|im_start|>system\nYou are a coding assistant.\n<|im_end|>\n<|im_start|>user\nWrite a Python function to reverse a list.\n<|im_end|>\n<|im_start|>assistant\n"
inputs = tokenizer(prompt, return_tensors="pt")
output = model.diffusion_generate(
    input_ids=inputs.input_ids.cuda(),
    max_new_tokens=100,
    steps=100,         # number of diffusion steps
    temperature=0.8,
    top_p=0.95
)
generated = tokenizer.decode(output.sequences[0], skip_special_tokens=True)
print(generated.split("<|dlm_pad|>")[0])  # remove padding token

    This mirrors the Hugging Face DiffuCoder example【41†L93-L101】. The model will start from a noisy full sequence and iteratively denoise it for 100 steps, outputting a complete function.

    Evaluation#

    To evaluate the diffusion model, use code benchmarks. Compute pass@k on HumanEval, MBPP, etc. The DiffuCoder paper uses EvalPlus (average of HumanEval+ and MBPP+ correctness)【39†L13-L16】. In practice, run your model’s generator many times per prompt and check test-case pass rates. Also measure generation speed: diffusion models often generate larger blocks in parallel, offering throughput improvements over AR decoding.

    Conclusion#

    By leveraging recent diffusion-LM tools and techniques, you can turn an AR code model like Qwen3-Coder-Next into a diffusion-based model. Key steps are: continual pretraining with a masked diffusion objective (using libraries like dLLM or Open-dLLM【18†L281-L290】【31†L524-L532】), followed by optional RL fine-tuning. This process is now well-documented by examples such as Apple’s DiffuCoder (based on Qwen2.5)【35†L252-L261】 and open-source projects【29†L346-L354】【23†L281-L290】. While full-scale training requires significant compute, the community-provided code and configs make small-scale experimentation and fine-tuning quite feasible.
